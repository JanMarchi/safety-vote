# Story 1.5: Role-Based Access Control (RBAC)

**Status:** Done
**Epic:** Epic 1 - Authentication & Authorization
**Story ID:** STORY-1.5
**Complexity:** 8 story points
**Priority:** High

---

## Summary

Extend authorization system with flexible role assignment, permission inheritance, and dynamic role management. Allow admins to assign roles to users, manage role-specific permissions, and create audit trails for permission changes. Implement permission caching for performance.

---

## Description

Building on Story 1.4 (API Authorization), this story enables flexible role management:

1. **Role Assignment** (admin assigns roles to users)
2. **Permission Inheritance** (rh inherits from eleitor privileges)
3. **Real-time Permission Updates** (changes apply immediately)
4. **Comprehensive Audit Logging** (all role changes tracked)
5. **Permission Caching** (5-minute cache with immediate invalidation)

---

## Acceptance Criteria

### AC1: Admin Can Assign Roles to Users
**Given:** Admin user authenticated
**When:** Admin calls PUT /api/admin/users/:id with `{ role: "rh" }`
**Then:** User's role updated in database, audit log recorded, response 200 OK

### AC2: User Permissions Reflect Assigned Role
**Given:** User assigned 'rh' role
**When:** User attempts rh-specific operation (e.g., manage users)
**Then:** Authorization check passes, operation succeeds

### AC3: Role Changes Are Immediate
**Given:** User with 'eleitor' role
**When:** Admin changes role to 'rh' via API
**Then:** Next request from user evaluates new permissions (no cache delay)

### AC4: Audit Log Tracks All Role Changes
**Given:** Admin changes user role from 'eleitor' to 'rh'
**When:** Change is applied
**Then:** Audit log records: admin_id, user_id, old_role, new_role, timestamp, change_reason

### AC5: Default Roles Exist (admin, rh, eleitor)
**Given:** System initialized
**When:** New user created
**Then:** User assigned one of three fixed roles (default: 'eleitor')

### AC6: Permission Inheritance Works
**Given:** 'rh' role inherits all 'eleitor' permissions
**When:** RH user attempts eleitor-level action (e.g., voting)
**Then:** Authorization passes (rh has all eleitor + rh-specific perms)

## Scope

### In Scope
- Role assignment API endpoints
- Permission inheritance logic (rh ⊇ eleitor)
- Role change audit logging
- Permission caching (Redis or in-memory)
- Admin role management interface
- Three fixed roles only (admin, rh, eleitor)

### Out of Scope
- Custom role creation (v2 feature)
- LDAP/Active Directory sync (future)
- Dynamic permission rules (future)
- Role expiration/temporary roles (future)
- Cross-tenant role management

## Risks

1. **Cache Staleness** — If cache isn't invalidated properly, users could retain old permissions
   - *Mitigation:* Immediate invalidation on role change + TTL fallback

2. **Audit Completeness** — If role change logging is incomplete, compliance violations
   - *Mitigation:* Database trigger + application logging (dual-layer)

3. **Permission Inheritance Bugs** — Incorrect inheritance could grant/deny wrong permissions
   - *Mitigation:* Comprehensive test coverage (AC6 specifically)
