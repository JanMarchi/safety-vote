# Story 1.4: API Authorization & Permissions

**Status:** Done
**Epic:** Epic 1 - Authentication & Authorization
**Story ID:** STORY-1.4
**Complexity:** 13 story points
**Priority:** High (Security Critical)

---

## Summary

Implement authorization checks on all API endpoints to enforce role-based permissions. Ensure only authenticated users with proper roles can access sensitive operations. Integrate with session middleware to verify user identity and authorization level on every request.

---

## Description

The Safety Vote system requires strict authorization controls at the API layer to prevent unauthorized access to sensitive operations. This story implements:

1. **Session-based authentication** (via Story 1.3 integration)
2. **Role-based permission checks** (admin, rh, eleitor roles)
3. **Per-endpoint authorization** (middleware + decorators)
4. **Authorization failure logging** (audit trail)
5. **Clear error responses** (401 vs 403 vs 500)

---

## Acceptance Criteria

### AC1: Unauthenticated Requests Are Rejected
**Given:** User without valid session token
**When:** User makes API request without Authorization header
**Then:** Endpoint returns 401 Unauthorized with message "Missing or invalid session"

### AC2: Expired/Revoked Sessions Are Blocked
**Given:** User with expired session token
**When:** User makes API request with expired token
**Then:** Session middleware rejects, returns 401 "Session expired"

### AC3: Role-Based Permission Check Works
**Given:** User with 'eleitor' role
**When:** User attempts to access admin-only endpoint (e.g., POST /api/elections)
**Then:** Endpoint returns 403 Forbidden with message "Insufficient permissions"

### AC4: RH Can Manage Users
**Given:** User with 'rh' (human resources) role
**When:** User calls POST /api/users or PUT /api/users/:id within same company
**Then:** Request succeeds, user created/updated in database

### AC5: Admin Can Access All Endpoints
**Given:** User with 'admin' role
**When:** User calls any API endpoint
**Then:** Authorization passes (role check succeeds)

### AC6: Authorization Failures Are Logged
**Given:** User attempts unauthorized action
**When:** Authorization check fails
**Then:** Audit log records: user_id, endpoint, action, timestamp, failure_reason

## Scope

### In Scope
- Session + permission validation middleware
- Per-endpoint authorization decorators/checks
- Error responses (401, 403, 500) with clear messages
- Audit logging for all auth failures
- Admin bypass option (with warning logs)
- Three fixed roles: admin, rh, eleitor

### Out of Scope
- Frontend permission UI/buttons
- OAuth/SSO integration (future)
- Multi-factor authentication (future)
- Custom role creation (fixed roles only in v1)
- Dynamic permission rules (future)

## Risks

1. **Incomplete Authorization** — If authorization checks are missing on any endpoint, unauthorized access possible
   - *Mitigation:* Comprehensive test coverage for all AC, security audit

2. **Performance Impact** — Authorization checks could slow down API responses
   - *Mitigation:* Implement permission caching, benchmark <100ms overhead

3. **Audit Logging Overhead** — Logging all failures could create performance bottleneck
   - *Mitigation:* Async logging, batch writes to audit_logs table
