## Description

This story implements the core Magic Link authentication system that allows users to log in securely without passwords. Users receive a unique, time-limited link via email that authenticates them when clicked.

### Problem Statement
Users need a secure, passwordless way to access Safety Vote. Magic Link authentication provides this by:
1. Eliminating weak password vulnerabilities
2. Reducing support burden for password resets
3. Providing a frictionless user experience
4. Supporting LGPD/GDPR compliance (no password storage)

### User Need
As a **user**, I want to **log in securely without remembering a password**, so that **I can access Safety Vote quickly and safely**.

### Solution Overview

The Magic Link flow works as follows:

```
1. User visits login page and enters email
2. Frontend validates email format
3. Frontend sends POST /api/auth/send-magic-link with email
4. Backend validates email:
   - Check rate limiting (max 5 attempts/10 min per IP)
   - Check if user exists or can be registered
   - Generate cryptographically secure token
   - Save token to database with 24h expiration
5. Backend sends email with magic link
6. User receives email and clicks link containing token
7. Frontend detects token in URL (?token=...)
8. Frontend sends POST /api/auth/verify-magic-link with token
9. Backend validates token:
   - Check token exists and hasn't expired
   - Check token hasn't been used already
   - Mark token as used
   - Create JWT session token
   - Create audit log entry
10. Backend returns JWT token to frontend
11. Frontend stores JWT in localStorage/sessionStorage
12. Frontend redirects to dashboard
```

### Key Features
- **Passwordless Security**: No passwords to steal, guess, or reset
- **Time-Limited Tokens**: Links expire after 24 hours
- **Single-Use Tokens**: Each link can only be used once
- **Rate Limiting**: Max 5 requests per 10 minutes per IP address
- **Email Validation**: Sends unique link via email
- **Session Creation**: Automatic JWT session after link validation
- **Audit Trail**: All attempts logged for security monitoring
- **Mobile Friendly**: Works seamlessly on all devices

- [x] Frontend extracts token from URL parameter

- [x] Backend validates token hasn't expired

### AC4: Rate limiting enforces max 5 attempts per 10 minutes per IP
**Given** a user/attacker attempts to request magic links
**When** they exceed 5 requests in a 10-minute window
**Then**:
- [x] System tracks requests by IP address
- [x] First 5 requests in 10 min window: allowed
- [x] 6th request in same window: blocked
- [x] Blocked request returns HTTP 429 (Too Many Requests)
- [x] Response includes `Retry-After` header
- [x] Response includes helpful error message
- [x] Error message shows when user can retry
- [x] Rate limit resets after 10 minutes without new requests
- [x] Audit log records "auth.rate_limit_exceeded" action
- [x] Rate limiting works per IP (not per email)
- [x] Different IPs get independent rate limit buckets

### AC5: Token validation prevents security vulnerabilities

**When** they try to:
- [x] Use expired token → Rejected with "Link expired"
- [x] Use already-used token → Rejected with "Link already used"
- [x] Modify token → Rejected with "Invalid token"
- [x] Use invalid token → Rejected with "Invalid token"
- [x] Reuse same link → Second attempt fails
- [x] Use someone else's token → Works only for that user

- [x] Error messages don't leak security info

## Technical Details

### Database Schema

#### magic_tokens table
```sql
CREATE TABLE magic_tokens (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  email VARCHAR(255) NOT NULL,
  token VARCHAR(256) UNIQUE NOT NULL,
  token_hash VARCHAR(64) UNIQUE NOT NULL, -- SHA-256 for lookup
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  company_id UUID NOT NULL REFERENCES companies(id),
  is_used BOOLEAN DEFAULT FALSE,
  used_at TIMESTAMP WITH TIME ZONE,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  expires_at TIMESTAMP WITH TIME ZONE NOT NULL, -- NOW() + 24h
  ip_address INET,
  user_agent TEXT,
  CONSTRAINT token_not_reused CHECK (
    (is_used = FALSE) OR (is_used = TRUE AND used_at IS NOT NULL)
  )
);

CREATE INDEX idx_magic_tokens_token_hash ON magic_tokens(token_hash);
CREATE INDEX idx_magic_tokens_email ON magic_tokens(email);
CREATE INDEX idx_magic_tokens_expires_at ON magic_tokens(expires_at);
CREATE INDEX idx_magic_tokens_user_id ON magic_tokens(user_id);
```

#### Rate Limiting Storage (Redis or in-memory)
```
Key: rate_limit:{ip_address}:{email}
Value: { count: number, reset_at: timestamp }
```

## Implementation Notes

### Security Best Practices
1. **Token Generation**: Use `crypto.randomBytes(32)` for 256-bit entropy
2. **Token Storage**: Never store plain token in database, only hash (SHA-256)
3. **Token Lookup**: Use hashed token for database queries
4. **Rate Limiting**: Implement with Redis or in-memory cache
5. **Email Security**: Don't log email addresses with tokens
6. **HTTPS Only**: All auth endpoints require HTTPS
7. **CSRF Protection**: Include CSRF token in form submission
8. **Input Validation**: Validate email on frontend and backend
9. **Error Messages**: Don't reveal user existence in error messages
10. **Audit Logging**: Log all attempts (success and failure)

### Performance Considerations
- Email sending should be asynchronous (queue-based)
- Database queries should use indexes on token_hash and email
- Rate limiting should use fast cache (Redis preferred)
- Token verification should complete in <100ms
- Consider caching user info after successful login
