## Title

## Description

This story implements secure session management with JWT token refresh, expiration, and logout functionality. Users can maintain secure persistent logins with automatic token refresh, preventing unexpected logouts while maintaining strict security controls.

### Problem Statement
After successful magic link authentication, users need a way to:
1. Stay logged in across browser sessions and page refreshes
2. Have tokens refresh automatically without requiring re-login
3. Logout cleanly and revoke their sessions
4. Access their account from multiple devices independently
5. Have sessions automatically expire for security

Without proper session management, users experience frustrating logouts, while the system remains vulnerable to token abuse.

### User Need
As a **Safety Vote user**, I want to **stay logged in securely without unexpected logouts and be able to logout when needed**, so that **I can work efficiently while maintaining control over my account security**.

### Solution Overview

The session management system works as follows:

```
1. User successfully verifies magic link (Story 1.1)
2. System creates user_sessions record with:
   - JWT access token (15 min expiration)
   - Refresh token (30 day expiration)
   - Device info (IP, user agent, fingerprint)
3. Frontend stores both tokens securely
4. On every request, frontend includes access token
5. When access token expires:
   - Frontend detects 401 response
   - Frontend calls /api/auth/refresh-token with refresh token
   - Backend validates refresh token and issues new access token
   - Frontend retries original request with new token
6. User calls /api/auth/logout:
   - Backend marks session as revoked
   - Frontend clears both tokens
   - User must re-authenticate to access system
7. Background cleanup job runs daily:
   - Deletes expired sessions (older than 24 hours of last activity)
   - Revokes abandoned sessions
```

### Key Features
- **Short-Lived Access Tokens**: 15-minute expiration for security
- **Long-Lived Refresh Tokens**: 30-day expiration for convenience
- **Sliding Window Sessions**: Expiration extends on activity
- **Multiple Device Support**: Track each device separately
- **Device Fingerprinting**: IP address + user agent hashing
- **Clean Logout**: Revoke single session without affecting other devices
- **Automatic Cleanup**: Expired sessions removed daily
- **Secure Storage**: No plaintext tokens in database
- **Session Revocation**: Invalidate compromised sessions immediately
- **Audit Trail**: All session events logged for security

## Technical Details

### Database Schema

#### user_sessions table
```sql
CREATE TABLE user_sessions (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  company_id UUID NOT NULL REFERENCES companies(id) ON DELETE CASCADE,
  access_token_hash VARCHAR(256) NOT NULL, -- SHA-256 hash of token
  refresh_token_hash VARCHAR(256) NOT NULL UNIQUE, -- SHA-256 hash of token
  ip_address INET,
  user_agent VARCHAR(500),
  device_fingerprint VARCHAR(256), -- SHA-256(ip_address + user_agent)
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  last_activity TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  expires_at TIMESTAMP WITH TIME ZONE NOT NULL, -- last_activity + 24h
  revoked_at TIMESTAMP WITH TIME ZONE NULL,
  is_active BOOLEAN GENERATED ALWAYS AS (revoked_at IS NULL) STORED,

  -- Foreign key constraints
  CONSTRAINT fk_user_sessions_user FOREIGN KEY (user_id) REFERENCES users(id),
  CONSTRAINT fk_user_sessions_company FOREIGN KEY (company_id) REFERENCES companies(id),

  -- Constraints
  CONSTRAINT valid_token_hashes CHECK (
    length(access_token_hash) > 0 AND length(refresh_token_hash) > 0
  ),
  CONSTRAINT valid_timestamps CHECK (
    created_at <= last_activity AND
    last_activity <= expires_at
  )
);

-- Indexes for performance
CREATE INDEX idx_user_sessions_user_id ON user_sessions(user_id);
CREATE INDEX idx_user_sessions_company_id ON user_sessions(company_id);
CREATE INDEX idx_user_sessions_refresh_token_hash ON user_sessions(refresh_token_hash);
CREATE INDEX idx_user_sessions_access_token_hash ON user_sessions(access_token_hash);
CREATE INDEX idx_user_sessions_expires_at ON user_sessions(expires_at);
CREATE INDEX idx_user_sessions_is_active ON user_sessions(is_active) WHERE revoked_at IS NULL;
CREATE INDEX idx_user_sessions_user_active ON user_sessions(user_id, is_active);

-- Enable Row Level Security (integrates with Story 1.2)
ALTER TABLE user_sessions ENABLE ROW LEVEL SECURITY;

CREATE POLICY "user_sessions_isolation" ON user_sessions
  USING (company_id = current_setting('app.current_company_id')::uuid);
```

### Token Expiration Strategy

**Access Token (JWT):**
- Expiration: 15 minutes from issuance
- Used for API requests (Authorization header)
- Short-lived for security (limits exposure if token leaked)
- Should be stored in memory or httpOnly cookie (not localStorage)

**Refresh Token:**
- Expiration: 30 days from issuance
- Stored in database (refresh_token_hash)
- Used to obtain new access tokens
- Should be stored in secure cookie (httpOnly, Secure, SameSite)

**Session Expiration (Sliding Window):**
- Base: 24 hours from last_activity
- Updates on every successful token refresh or API request
- Prevents logout during active use
- User inactive for 24+ hours = automatic session expiration

## Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|-----------|
| **Token leakage in localStorage** | Medium | Critical | Document security considerations, recommend httpOnly cookies, implement CSP headers |
| **Refresh token abuse/brute force** | Medium | High | Rate limit refresh endpoint (10 req/min per IP), track token usage, rotation on use |
| **Session hijacking via IP spoofing** | Low | High | Use device fingerprinting, validate user agent consistency, log suspicious patterns |
| **Token stored in plaintext** | Low | Critical | Hash all tokens before database storage, never log full tokens |
| **Refresh token doesn't rotate** | Medium | Medium | Optional: Rotate refresh token on each use, implement refresh token family tracking |
| **Cleanup job doesn't run** | Low | Medium | Schedule cleanup job in CI/CD, add alerting if job fails, document manual cleanup |
| **Session expiration too aggressive** | Medium | Medium | Use 24-hour sliding window, test with realistic user workflows |
| **Concurrent token refresh race condition** | Low | Medium | Use database transactions, versioning, or atomic operations for token updates |

## Implementation Notes

### Security Best Practices
1. **Token Storage**: Hash all tokens before database storage (SHA-256 minimum)
2. **Token Transmission**: Always use HTTPS, never log full tokens
3. **Frontend Storage**: Prefer httpOnly cookies over localStorage
4. **Token Validation**: Check expiration, signature, and revocation status
5. **Refresh Limiting**: Rate limit refresh endpoint to prevent abuse
6. **Device Fingerprinting**: Use IP + user agent for basic device tracking
7. **Audit Trail**: Log all session operations with timestamp and user
8. **Secure Cleanup**: Don't leak session data during deletion

### Performance Considerations
- Access token refresh should complete in <50ms
- Session lookup should use indexed queries
- Cleanup job should run off-peak (2 AM UTC)
- Cache frequently accessed sessions (optional)
- Batch cleanup operations for large datasets

### Token Refresh Flow
```
Frontend Request → API Middleware:
  1. Extract access token from Authorization header
  2. Validate token signature
  3. Check token hasn't expired
  4. Load session from database using session_id in token
  5. Verify session not revoked
  6. Update last_activity timestamp
  7. Proceed with request

If Token Expired:
  1. Return 401 Unauthorized
  2. Frontend retries with refresh_token
  3. Backend validates refresh token
  4. Backend checks session exists and not revoked
  5. Backend issues new access token
  6. Return new token to frontend
  7. Frontend retries original request with new token
```
