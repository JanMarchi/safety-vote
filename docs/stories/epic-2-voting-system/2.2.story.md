# Story 2.2: Candidate Registration & Management

**Status:** Done
**Epic:** Epic 2 - Voting System & Election Management
**Story ID:** STORY-2.2
**Complexity:** 9 story points
**Priority:** High

---

## Summary

Implement candidate registration and management for elections. Allow users to register as candidates in draft elections, and allow admins/RH to manage candidate lists (add, remove, approve/reject candidates).

---

## Acceptance Criteria

### AC1: User Can Register As Candidate
**Given:** Eleitor user, election in draft status
**When:** User calls POST /api/candidates with election_id
**Then:** Candidate registered, returns 201, user appears as candidate

### AC2: Candidate Can't Register in Active Election
**Given:** Election status is 'active'
**When:** User tries to register as candidate
**Then:** Returns 403 Forbidden with message "Cannot register after election starts"

### AC3: One Candidate Per Election Per User
**Given:** User already registered as candidate in election
**When:** User tries to register again
**Then:** Returns 409 Conflict with message "Already registered as candidate"

### AC4: RH Can Add Candidates
**Given:** RH user
**When:** RH calls POST /api/candidates with candidate data
**Then:** Candidate created, can set name + proposal

### AC5: RH Can Remove Candidates
**Given:** RH user, election in draft
**When:** RH calls DELETE /api/candidates/:id
**Then:** Candidate deleted, returns 200

### AC6: Candidates Isolated by Election Company
**Given:** User from Company A
**When:** User accesses Company B's election candidates
**Then:** Returns 403 Forbidden

---

## Scope

### In Scope
- Candidate self-registration (user registers as candidate)
- Candidate management (RH add/remove)
- Candidate info (name, proposal, photo_url)
- Candidate listing by election
- Self-deregistration (before election starts)

### Out of Scope
- Candidate approval workflow
- Photo upload to cloud storage
- Candidate ranking/vetting
- Multiple candidates per user
